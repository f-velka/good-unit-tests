# 4種類のコード分類と Humble Object パターン

対象コードを「複雑さ・ドメイン上の重要性」と「協力者（依存オブジェクト）の数」の2軸で分類する。

| | 協力者が少ない | 協力者が多い |
|---|---|---|
| **複雑さ/ドメイン重要性が高い** | ドメインモデル・アルゴリズム | 過度に複雑なコード |
| **複雑さ/ドメイン重要性が低い** | 取るに足らないコード | コントローラ |

## ドメインモデル・アルゴリズム

- 単体テストを最も手厚く書くべき対象
- 協力者が少ない（＝in-processな依存のみに閉じている）ため、テストが速く、リファクタリング耐性も保ちやすい

## 取るに足らないコード

- 引数なしコンストラクタ、1行だけの getter/setter、単純な委譲など
- テストを書いても退行に対する保護がほぼ得られない一方、保守コストだけがかかる
- **テストしない**（このコードにテストを書くこと自体がアンチパターン）

## コントローラ

- 外部依存（DB、外部API、他サービスなど）との調整役。ロジックは薄いが協力者が多い
- 単体テストではなく、実際の依存（あるいはそれに近いテスト用インスタンス）を使った**統合テスト**で
  少数のケースを広くカバーする方針にする

## 過度に複雑なコード（複雑さも協力者も多い）

- そのままでは単体テストが書きにくい（モックだらけになる、テストが実装に密結合する）
- **Humble Object パターン**で分割する:
  1. ロジック・分岐・計算を「ドメインモデル」側に抽出する（協力者は少ないまま、深いロジックを持たせる = 狭いが深い）
  2. 残った部分は依存を呼び出すだけの「謙虚 (humble) な」薄いラッパー（＝コントローラ）にする（広いが浅い）
  3. ドメインモデル側を単体テストで手厚くカバーし、コントローラ側は統合テストで薄くカバーする

## 実務での適用

- 新しいコードを書くときは、最初からロジックをドメインモデルに、外部連携をコントローラに分離する設計を意識する
- 既存の「過度に複雑なコード」に対してテストを追加・改善する依頼が来た場合は、テストを増やす前に
  Humble Object パターンでの分割をまず提案する
