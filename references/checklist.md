# 仕上げチェックリスト（アンチパターン照合）

テストを書き終えたら、以下のそれぞれについて該当しないか確認する。

## 構造・スコープ

- [ ] private なメソッド/フィールドを直接テストしていないか（公開APIを通じた間接テストになっているか）
- [ ] テストのためだけにカプセル化を破っていないか（テスト専用の public getter を実装コードに追加していないか）
- [ ] 1テストに Act が2回以上ないか、AAA ブロックが複数回繰り返されていないか（→ テスト分割）
- [ ] テスト本体に `if`/`for`/`while` などの分岐・ループがないか（テストのサイクロマティック複雑度は0に保つ）
- [ ] 1つのプロダクションクラスに対して機械的に1つのテストクラスを対応させていないか
      （テストは「振る舞い」を単位に構成する。1クラスに複数の振る舞いがあれば複数の観点でテストが分かれてよいし、
      1つの振る舞いが複数クラスにまたがってもよい）

## 検証内容

- [ ] アサーションは「観察可能な振る舞い（公開された操作・状態）」に対してのみ行っているか
- [ ] 実装アルゴリズムをテストコード側で再実装して比較する形になっていないか
      （＝ドメイン知識の漏洩。実装が変わるたびにテスト側のロジックも追随して直す必要が生まれ、
      退行に対する保護がほぼ0になる典型的な失敗パターン）
- [ ] モックの呼び出し回数・順序・引数を、検証に必要な範囲を超えて厳密に固定していないか（過剰仕様化）
- [ ] 管理下にある依存（自社DBなど）をモックにしていないか

## Arrange の整理

- [ ] Arrange が複雑で読みにくくなっていないか。複雑ならテストデータビルダー（Object Mother/Builder パターン）で
      「そのテストで重要な値だけを明示し、残りはデフォルト値に任せる」形に整理する
- [ ] 複数のテストで同じ Arrange ロジックを重複して書いていないか（共通化はしてよいが、
      共有ロジックの変更が一部のテストの意図を見えなくする「Mystery Guest」状態にはしない。
      テストを読むだけでそのテストの前提が分かる状態を保つ）

## 命名

- [ ] `MethodName_Scenario_ExpectedResult` のような機械的フォーマットに縛られていないか
- [ ] テスト名がそのドメインを知らない人にも読める自然文の説明になっているか
      （例: `delivery_with_a_past_date_is_invalid`）

## テストする価値があるか

- [ ] 取るに足らないコード（getter/setter、引数なしコンストラクタ、単純な委譲）にテストを書いていないか
- [ ] 過度に複雑なコードに対して、分割せずにモックを積み上げてテストしようとしていないか
      （→ Humble Object パターンでドメインモデルとコントローラに分割することを先に検討する）

## 最終確認の問い

- このテストは、実装だけを別の方法に書き換えても（振る舞いが同じなら）通り続けるか？
- このテストが失敗したら、それは実際にバグがあることを意味するか？（偽陽性でないか）
- 半年後、このコードを知らない人がこのテストだけを読んで、何を保証しているか理解できるか？
